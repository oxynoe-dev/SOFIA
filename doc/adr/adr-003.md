# ADR-003 — Stratégie d'internationalisation

**Statut** : Proposed
**Date** : 2026-03-30
**Auteur** : Mira

## Contexte

Voix est un produit open source à vocation internationale (GitHub, communauté dev). Le contenu existant est majoritairement en français (méthode, exemples Katen, retours). Le public cible est mondial.

L'i18n dans un repo git de documentation a un coût structurel : duplication de contenu, synchronisation des traductions, choix de la langue source, impact sur les contributions.

### Options

**A — Anglais first, français en override**
- `methode/`, `claude-code/`, `outillage/` en anglais (langue source)
- `i18n/fr/` contient les traductions françaises (même arborescence)
- Une seule source de vérité, l'anglais
- Le français existant doit être traduit → effort initial significatif

**B — Français first, anglais en override**
- Contenu existant reste en français (pas de migration)
- `i18n/en/` contient les traductions anglaises
- Moins d'effort initial, mais le français comme source limite les contributions internationales

**C — Dossiers par langue à la racine**
- `en/methode/`, `fr/methode/`
- Standard (React docs, Kubernetes docs, Docusaurus)
- Duplication d'arborescence, sync manuelle entre les deux
- Fonctionne bien avec un tooling de doc (build.py, site statique)

**D — Suffixe fichier**
- `artefacts.fr.md`, `artefacts.en.md` côte à côte
- Diff facile, pas de duplication d'arbo
- Arborescence bruyante au-delà de 2 langues

## Décision

À trancher. Points à peser :

- **Portée** : combien de langues à terme ? 2 (fr/en) ou plus ?
- **Contribution** : veut-on des contributions externes en anglais dès le départ ?
- **Voix du PO** : la méthode porte une voix personnelle — la traduction est-elle un risque de dilution ?
- **Timing** : i18n avant ou après la v1.0.0 ?
