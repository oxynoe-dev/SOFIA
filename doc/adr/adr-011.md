# ADR-011 — Protocole H2A (Human-to-Assistant)

**Statut** : Accepted
**Date** : 2026-04-16
**Auteur** : Aurele

## Contexte

Jusqu'a la v0.3.1, la coordination humain-assistants etait decrite de maniere diffuse : conventions dans protocol/, principes dans core/, regles operationnelles dans la documentation. Il n'existait pas de formalisation unifiee du protocole de coordination.

Trois constats ont motive la formalisation :

1. **Positionnement manquant.** MCP (Anthropic) couvre agent-outils, A2A (Google) couvre agent-agent. La couche humain-assistant n'avait pas de nom ni de spec.

2. **Friction sous-specifiee.** Les marqueurs existaient mais sans filiation theorique, sans resolutions formelles, sans mecanisme de mitigation de l'opacite de l'orchestrateur.

3. **Auditabilite ambigue.** Pas de distinction claire entre ce qu'un outil d'audit peut verifier mecaniquement (protocolaire) et ce qui releve du jugement (observationnel).

## Decision

Formaliser H2A comme protocole organisationnel dans `protocol/h2a.md`, avec des documents satellites (`friction.md`, `exchange.md`, `contribution.md`).

### Structure du protocole

**5 invariants** — proprietes constitutives sans lesquelles H2A n'est plus H2A :
1. Friction constitutive
2. Humain arbitre
3. Isolation
4. Tracabilite
5. Opacite residuelle (limitation structurelle, pas une capacite)

**7 entites** — Instance, Espace, Persona, Orchestrateur, Echange, Friction, Contribution. Modele conceptuel dans `core/modele.md`.

**9 operations** — de `ouvrirSession()` a `signalerPattern()`. Definies en semantique, pas en wire format.

**2 couches de formalisation** :
- Protocolaire — verification computationnelle, deterministe, automatisable
- Observationnelle — verification inferentielle, jugement semantique, best-effort

### Elements cles

**Resolutions PXP** — 4 gestes epistemiques (`ratifie`, `conteste`, `revise`, `rejete`) qualifiant le denouement de chaque friction. Inspire de Mestha et al. (2025).

**signalerPattern()** — operation de mitigation de l'opacite de l'orchestrateur. Detection de convergence thematique de rejets, 3 hypotheses argumentees, qualification obligatoire avec asymetrie de charge de preuve.

**Mutabilite inter-sessions** — une resolution peut evoluer dans une session ulterieure, avec reference tracable (`ref:`).

**Filiation principes → invariants** — les invariants derivent des principes (core/) mais ne les couvrent pas tous. Les principes guident la methode entiere, les invariants ne portent que sur le protocole.

### Terminologie

Mots-cles au sens RFC 2119 (DOIT, DEVRAIT, PEUT).

## Alternatives considerees

1. **Pas de formalisation** — Garder les conventions diffuses. Rejete : l'audit et le multi-instance exigent une spec stable.

2. **Protocole technique d'emblee** — Definir un wire format (JSON, endpoints). Rejete : premature. Le protocole est organisationnel, l'implementation technique viendra quand le besoin sera reel.

3. **Fusion avec le core** — Mettre les invariants H2A dans core/principes.md. Rejete : les invariants portent sur le protocole d'echange, pas sur la methode entiere.

## Consequences

- `protocol/h2a.md` devient le document de reference pour la coordination
- `protocol/friction.md`, `exchange.md`, `contribution.md` specifient les entites
- L'audit est fonde sur la distinction protocolaire/observationnelle
- Le document complet (positionnement, discussion, questions ouvertes) est dans `doc/h2a-proposal.md`
- Les futures implementations (API, BDD) peuvent etre conformes a H2A tant qu'elles respectent la semantique

Refonte v0.3.2 : convergence entre la formalisation des principes (core/), la revue de l'article SOFIA (note-equipe-reflexions-h2a.md), et le besoin d'auditabilite multi-instance.
