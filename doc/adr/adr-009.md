# ADR-009 — Couche Instance et position de Sofia

**Statut** : Proposed
**Date** : 2026-04-10
**Auteur** : Mira
**Supersède partiellement** : ADR-008 (ajoute une couche, ne remplace pas les trois existantes)

## Contexte

L'ADR-008 a structuré le repo en trois couches : Core (invariants), Protocol (contrat d'interface), Runtime (implémentation). L'alternative "quatre couches" avait été rejetée — le besoin n'était pas clair.

Deux constats issus des sessions Rodin (09-10/04) changent la donne :

**1. Les personas d'instance ne sont pas dans le core.** Les archétypes (architecte, dev, stratège...) sont dans le core — ce sont des modèles réutilisables. Mais les personas concrets d'une instance (Mira, Axel, Marc...) sont propres à l'orchestrateur. Ils portent un calibrage spécifique, des circuits de friction spécifiques, une topologie spécifique. Ce n'est ni du core, ni du protocol, ni du runtime.

**2. Sofia doit être hors de l'instance pour l'auditer objectivement.** Sofia vit actuellement dans le produit (core/sofia.md) ET dans l'instance (workspace experiments/sofia/). C'est un conflit structurel : elle est juge et partie. Le même pattern que Rodin s'applique — l'extériorité est la condition de l'objectivité.

## Décision

### Quatre couches

```
sofia/
├── core/              ← invariants (principes, friction, devoirs, personas, templates archétypes)
├── protocol/          ← contrat d'interface (artefacts, conventions, tracabilité, isolation, orchestration)
├── runtime/           ← implémentation (claude-code)
└── doc/               ← documentation, terrain, décisions
```

La couche **Instance** n'est pas dans le repo produit — elle est dans chaque instance déployée :

```
instance/
├── sofia.md           ← marqueur d'instance
├── shared/            ← bus + roadmaps + conventions d'instance
│   └── orga/          ← personas concrets, team-orga, circuits de friction
├── {workspace}/       ← un par persona concret
└── ...
```

La distinction : le produit sofia/ fournit les **archétypes et les règles**. L'instance fournit les **personas concrets et la topologie**.

### Position de Sofia

Sofia reste dans le produit (`core/sofia.md`). Elle n'a pas de workspace dans les instances. Elle intervient de l'extérieur pour :
- Installer une nouvelle instance (onboarding)
- Auditer une instance existante (vérification conformité)
- Monter une instance d'audit (challengers miroir)

Son extériorité est structurelle, pas accidentelle.

### Deux types d'instance

| | Instance opérationnelle | Instance d'audit |
|---|---|---|
| **Durée de vie** | Longue — suit le projet | Éphémère — détruite après synthèse |
| **Mémoire** | Cumulative (résumés de session) | Aucune entre cycles |
| **Personas** | Opérationnels, dans le flux | Challengers miroir 1:1 |
| **Produit** | Des livrables métier | Une synthèse d'audit |
| **Qui la monte** | Sofia (onboarding) | Sofia (à chaque version majeure) |

## Alternatives considérées

1. **Garder 3 couches, mettre Instance dans Protocol** — Protocol définit le contrat d'interface, pas le contenu d'instance. Mélanger les deux casse la portabilité : un même protocol peut servir des instances très différentes.

2. **Ajouter instance/ dans le repo produit** — Le repo produit fournit des templates et des archétypes, pas des personas concrets. Mettre une instance de référence dans le produit confond le modèle et l'exemple.

3. **Laisser Sofia dans l'instance** — Conflit d'intérêt structurel documenté. Rejeté.

## Conséquences

- La doc d'architecture (arch-sofia.md) passe de 3 à 4 couches conceptuelles (le repo produit reste à 3 répertoires — la 4e couche est dans les instances)
- Sofia n'a plus de workspace dans experiments/ (migration dans roadmap-experiments v0.3.0)
- Le concept d'instance d'audit est formalisé comme un type d'instance à part entière
- Les templates d'instance (sofia-instance.md, team-orga.md) restent dans protocol/templates/ — ils définissent la structure, pas le contenu

## Origine

Sessions Rodin (09-10/04) : identification du persona méta (hors flux, sans mémoire), fusion Nora×Luz (personas d'instance ≠ archétypes), condition cachée (le trait cognitif de l'orchestrateur précède la méthode). Note Sofia sur l'instance d'audit (10/04) : signal de circularité et proposition de séquencement.
