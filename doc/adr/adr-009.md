# ADR-009 — Couche Instance, restructuration repo et position de Sofia

**Statut** : Proposed
**Date** : 2026-04-10
**Auteur** : Mira
**Supersède partiellement** : ADR-008 (ajoute une couche, restructure le repo)

## Contexte

L'ADR-008 a structuré le repo en trois couches : Core (invariants), Protocol (contrat d'interface), Runtime (implémentation). L'alternative "quatre couches" avait été rejetée — le besoin n'était pas clair.

Trois constats changent la donne :

**1. Les personas d'instance ne sont pas dans le core.** Les archétypes (architecte, dev, stratège...) sont des modèles de référence. Les personas concrets d'une instance (Mira, Axel, Marc...) sont propres à l'orchestrateur. Ce n'est ni du core, ni du protocol, ni du runtime.

**2. Les archétypes ne sont pas des invariants.** Un orchestrateur construit ses personas depuis ses axes de tension, pas en copiant un template. Les archétypes montrent à quoi ressemble un persona par rôle — ils inspirent, ils ne scaffoldent pas. Ce n'est pas du core.

**3. Sofia doit être hors de l'instance pour l'auditer objectivement.** Sofia vit dans le produit (core/sofia.md) ET dans l'instance (workspace experiments/sofia/). C'est un conflit structurel : elle est juge et partie. Le même pattern que Rodin s'applique — l'extériorité est la condition de l'objectivité.

## Décision

### Quatre couches dans le repo produit

```
sofia/
├── core/              ← invariants (principes, friction, devoirs, anatomie persona, sofia)
├── protocol/          ← les règles (artefacts, conventions, tracabilité, isolation, orchestration)
├── instance/          ← références pour construire une instance
│   ├── archetypes/    ← modèles de personas par rôle (pour s'en inspirer, pas pour copier)
│   ├── artefacts/     ← à quoi ressemble une note, une review, une session...
│   └── examples/      ← katen/ (comment ça se manifeste sur le terrain)
├── runtime/           ← implémentation (claude-code)
└── doc/               ← documentation, feedback, patterns, ADR
```

**Core** = ce qui ne bouge jamais. Les principes, la friction, les devoirs, l'anatomie d'un persona. Si on change ça, on change la méthode.

**Protocol** = les règles qu'une instance doit respecter. Conventions, nommage, isolation, orchestration. Le contrat.

**Instance** = les références pour construire sa propre instance. Archétypes de personas, artefacts de référence (à quoi ressemble une note, une review, une session), et un exemple terrain complet (Katen — snapshot d'instance vivante avec shared/, workspaces, conventions). On s'en inspire, on ne copie pas.

**Runtime** = l'implémentation pour un provider donné. Remplaçable sans toucher au reste.

### Position de Sofia

Sofia reste dans le produit (`core/sofia.md`). Elle n'a pas de workspace dans les instances. Elle intervient de l'extérieur pour :
- Installer une nouvelle instance (onboarding)
- Auditer une instance existante (vérification conformité)
- Monter une instance d'audit (challengers miroir)

### Deux types d'instance

| | Instance opérationnelle | Instance d'audit |
|---|---|---|
| **Durée de vie** | Longue — suit le projet | Éphémère — détruite après synthèse |
| **Mémoire** | Cumulative (résumés de session) | Aucune entre cycles |
| **Personas** | Opérationnels, dans le flux | Challengers miroir 1:1 (noms inversés) |
| **Produit** | Des livrables métier | Une synthèse d'audit |
| **Qui la monte** | Sofia (onboarding) | Sofia (à chaque version majeure) |

## Alternatives considérées

1. **Garder 3 couches, pas d'instance/ dans le repo** — Les archétypes restent dans core/templates/, les exemples dans doc/examples/. La frontière "invariant vs référence d'instanciation" reste implicite. Rejeté : la confusion a déjà causé des erreurs de placement.

2. **instance/ comme kit de scaffolding (templates à copier)** — Nommer le répertoire "templates" et présenter les fichiers comme des modèles à copier-coller. Rejeté : SOFIA est une méthode à internaliser, pas un kit à déployer mécaniquement. Les archétypes et les formats sont des références — on s'en inspire, on construit le sien.

3. **Laisser Sofia dans l'instance** — Conflit d'intérêt structurel documenté. Rejeté.

4. **Mettre une instance vivante dans le repo produit** — Confond le modèle et le déploiement. `instance/examples/katen/` est un snapshot qui montre la structure complète d'une instance, mais ne vit pas — la vraie instance vit dans son propre repo et évolue indépendamment.

## Conséquences

- La structure du repo change — liens internes à mettre à jour
- `core/templates/` et `protocol/templates/` disparaissent
- arch-sofia.md passe de 3 à 4 couches dans le repo
- Sofia n'a plus de workspace dans experiments/
- Le concept d'instance d'audit est formalisé
- Les contributeurs savent immédiatement où chercher : invariants → core/, règles → protocol/, références d'instanciation → instance/, implémentation → runtime/
- Le mental model change : on ne "déploie" pas SOFIA, on la comprend et on construit son instance
