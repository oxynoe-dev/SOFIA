# ADR-010 — Multi-plateforme via couche implementation

**Statut** : Accepted
**Date** : 2026-04-16
**Auteur** : Aurele
**Supersede** : ADR-004

## Contexte

L'ADR-004 prevoyait des repos separes par plateforme (`sofia-mistral`, `sofia-gemini`, etc.), chacun referencant le repo `sofia` pour les principes generiques.

Entre-temps, la v0.3.2 a introduit deux evolutions qui rendent cette approche obsolete :

1. **La couche `implementation/`** (cf. ADR-012) separe la semantique du protocole (protocol/) de son implementation concrete (implementation/filesystem/). Ce qui etait prevu comme un mapping externe dans un repo d'adaptation est maintenant une couche interne du repo.

2. **La couche `runtime/`** accueille deja l'implementation Claude Code dans `runtime/claude-code/`. La structure est prete pour `runtime/mistral/`, `runtime/gemini/` etc. sans restructuration.

La strategie "repos separes" disperse la methode et complique la coherence. L'alternative est de garder tout dans le meme repo, avec des adaptateurs runtime.

## Decision

Multi-plateforme dans le meme repo, pas en repos separes.

```
sofia/
├── core/              ← invariants (identique quel que soit le provider)
├── protocol/          ← semantique H2A (identique)
├── implementation/    ← implementation(s) du protocole
│   └── filesystem/    ← implementation courante
├── runtime/           ← adaptateurs par provider
│   ├── claude-code/   ← implementation de reference
│   ├── mistral/       ← adaptateur Mistral (cible v0.4)
│   └── ...
├── instance/          ← references d'instanciation
├── canvas/            ← outils d'inspiration
└── doc/
```

**Core + Protocol** = le noyau portable. Ne mentionne aucun provider.

**Implementation** = comment le protocole se materialise (filesystem aujourd'hui, API/BDD demain). Potentiellement multiples implementations.

**Runtime** = comment un provider specifique execute les instructions (CLAUDE.md, MISTRAL.md, hooks, memoire). Un adaptateur par provider.

La distinction implementation/runtime est la cle :
- `implementation/` repond a "comment les artefacts sont stockes et audites"
- `runtime/` repond a "comment le persona recoit ses instructions et persiste son contexte"

## Alternatives considerees

1. **Repos separes (ADR-004)** — Disperse la methode, complique la coherence inter-providers, cree des problemes de versionnage croise. Supersede.

2. **Tout dans runtime/** — Fusionner implementation et runtime dans un seul repertoire par provider. Rejete : la distinction semantique est reelle (un filesystem est partage entre providers, un CLAUDE.md ne l'est pas).

## Consequences

- Les repos `sofia-mistral`, `sofia-gemini` ne seront pas crees
- L'ADR-004 est supersede
- Chaque adaptateur runtime est autonome (un orchestrateur peut mixer les providers par persona)
- Le noyau portable (core/ + protocol/) reste la reference unique
- La v0.4 (multi-plateforme) se fait par ajout d'un `runtime/mistral/`, pas par creation d'un repo

## Origine

Emergence organique lors de la refonte v0.3.2 : l'extraction de `implementation/` et la formalisation H2A ont rendu explicite la structure qui rend les repos separes inutiles.
