# ADR-010 — Multi-platform via implementation layer

**Status**: Accepted
**Date**: 2026-04-16
**Author**: Aurele
**Supersedes**: ADR-004

## Context

ADR-004 planned separate repos per platform (`sofia-mistral`, `sofia-gemini`, etc.), each referencing the `sofia` repo for generic principles.

In the meantime, v0.3.2 introduced two evolutions that make this approach obsolete:

1. **The `implementation/` layer** (cf. ADR-012) separates the protocol semantics (protocol/) from its concrete implementation (implementation/filesystem/). What was planned as an external mapping in an adaptation repo is now an internal layer of the repo.

2. **The `runtime/` layer** already hosts the Claude Code implementation in `runtime/claude-code/`. The structure is ready for `runtime/mistral/`, `runtime/gemini/` etc. without restructuring.

The "separate repos" strategy scatters the method and complicates coherence. The alternative is to keep everything in the same repo, with runtime adapters.

## Decision

Multi-platform in the same repo, not in separate repos.

```
sofia/
├── core/              ← invariants (identical regardless of provider)
├── protocol/          ← H2A semantics (identical)
├── implementation/    ← protocol implementation(s)
│   └── filesystem/    ← current implementation
├── runtime/           ← adapters per provider
│   ├── claude-code/   ← reference implementation
│   ├── mistral/       ← Mistral adapter (target v0.4)
│   └── ...
├── instance/          ← instanciation references
├── canvas/            ← inspiration tools
└── doc/
```

**Core + Protocol** = the portable kernel. Mentions no provider.

**Implementation** = how the protocol materializes (filesystem today, API/DB tomorrow). Potentially multiple implementations.

**Runtime** = how a specific provider executes instructions (CLAUDE.md, MISTRAL.md, hooks, memory). One adapter per provider.

The implementation/runtime distinction is the key:
- `implementation/` answers "how artifacts are stored and audited"
- `runtime/` answers "how the persona receives its instructions and persists its context"

## Alternatives considered

1. **Separate repos (ADR-004)** — Scatters the method, complicates inter-provider coherence, creates cross-versioning problems. Superseded.

2. **Everything in runtime/** — Merge implementation and runtime into a single directory per provider. Rejected: the semantic distinction is real (a filesystem is shared between providers, a CLAUDE.md is not).

## Consequences

- The `sofia-mistral`, `sofia-gemini` repos will not be created
- ADR-004 is superseded
- Each runtime adapter is autonomous (an orchestrator can mix providers per persona)
- The portable kernel (core/ + protocol/) remains the single reference
- v0.4 (multi-platform) is done by adding a `runtime/mistral/`, not by creating a repo

## Origin

Organic emergence during the v0.3.2 overhaul: the extraction of `implementation/` and the H2A formalization made explicit the structure that renders separate repos unnecessary.
