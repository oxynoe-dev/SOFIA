# ADR-012 — Extraction de la couche implementation

**Statut** : Accepted
**Date** : 2026-04-16
**Auteur** : Aurele

## Contexte

Jusqu'a la v0.3.1, les outils d'implementation (audit-instance.py, create-instance.py, tests) vivaient dans `protocol/tools/`. Les conventions de rendu (comment ecrire une friction en Markdown, comment structurer un resume de session) etaient melangees avec la semantique du protocole dans `protocol/`.

Ce melange posait deux problemes :

1. **Confusion semantique/implementation.** "La friction porte 5 dimensions" (protocole) et "la friction est une ligne Markdown avec symbole + mot-cle + initiative" (implementation) cohabitaient dans les memes documents. Impossible de savoir ce qui est portable et ce qui est lie au filesystem.

2. **Blocage multi-implementation.** Si H2A est implemente demain via une API REST ou une base de donnees, les outils et les conventions de rendu changent, mais la semantique ne change pas. Avoir tout dans protocol/ oblige a toucher le protocole pour changer l'implementation.

## Decision

Extraire une couche `implementation/` separee du protocole.

```
sofia/
├── protocol/           ← semantique (quoi)
│   ├── h2a.md          ← invariants, operations, couches
│   ├── friction.md     ← dimensions, marqueurs, resolutions
│   ├── exchange.md     ← artefacts, sessions, frontmatter
│   └── contribution.md ← direction, types
├── implementation/     ← materialisation (comment)
│   ├── implementation.md  ← stack, conventions de rendu, mapping operations
│   └── filesystem/        ← outils pour l'implementation courante
│       ├── audit-instance.py
│       ├── create-instance.py
│       ├── conventions.md (template)
│       └── tests/
└── ...
```

**Protocol** repond a "qu'est-ce qu'une friction, qu'est-ce qu'une session, qu'est-ce qu'un artefact" — en termes de semantique, pas de format.

**Implementation** repond a "comment ca se materialise" — Markdown + frontmatter + git + repertoires. C'est la que vivent les conventions de rendu, le mapping des operations sur des gestes concrets, et les outils d'audit/scaffolding.

Le document `implementation/implementation.md` contient une table explicite "protocole vs implementation" pour chaque element.

## Alternatives considerees

1. **Tout dans protocol/** — Simple mais confond deux niveaux. Quand on ecrit "la friction est une ligne Markdown", on fait un choix d'implementation, pas de protocole. Rejete.

2. **Outils dans un repertoire tools/ a la racine** — Separe les scripts mais pas les conventions de rendu. Rejete : le probleme est conceptuel, pas juste organisationnel.

3. **Un repertoire par implementation (filesystem/, api/, bdd/)** — Premature. L'implementation filesystem est la seule qui existe. La structure `implementation/filesystem/` est prete pour accueillir d'autres implementations le jour ou elles seront reelles.

## Consequences

- `protocol/` ne contient plus de conventions de rendu ni de scripts
- `implementation/implementation.md` est le document de reference pour l'implementation courante
- Les outils (audit, create-instance) vivent dans `implementation/filesystem/`
- Le tag `implementation:filesystem` sur les canvas artefacts distingue ce qui est lie a l'implementation
- Une future implementation alternative (API, BDD) vivrait dans `implementation/api/` etc. sans toucher au protocole
- L'ADR-008 (Core/Protocol/Runtime) reste valide — implementation/ est une couche complementaire, pas un remplacement

## Origine

Refonte v0.3.2 : la formalisation de H2A (ADR-011) a rendu visible le melange semantique/implementation dans protocol/. La separation est devenue necessaire pour que le protocole reste portable.
